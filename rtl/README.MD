RTL design
===

The design is composed of multiple modules and files, including the following main files:
  - The top level design containing the entire array of PEs and NoC routers. (in `mocarabe.v`)
  - The PE module (in `pe_1_input.v` and `pe_2_input.sv`)
  - SRLs inside PEs (to store incoming operands)(in `pe_srl.sv`)
  - Multiplexers to fan-in the NoC output to PE inputs (in `pe_mux_2_input.sv` and `pe_mux_3_input.sv`)
  - NoC router module (in `torus_switch.sv`)
  - Various memory files to initialize CTX memories.

**Mocarabe.v**

This module contains the top level design, written in parametric Verilog. It instantiates PEs, torus routers, and fan-in muxes (if necessary) and connects them together to form the Mocarabe CGRA.

**PE**

This module contains the PE code. It can have 1-2 input ports to receive the incoming data, with the respecting modules called `pe_1_input.v` and `pe_2_input.v`. The incoming data comes from the torus routers, and is stored in shift(`PE_srl`) registers. At each cycle, the PE selects the relevant operands (based on the context memories generated by compiler) and performs addition/multiplication on them. The ALU is pipelined to increase clock frequency. The resulting data is sent to all torus routers connected to the PE.

Here is the PE modules' interface:

`pe_1_input.sv` (the 2 input pe just has an extra "i_operand" port):
```verilog
  module pe_1_input #(
      parameter SCHED_LEN = 16,//schedule length
      parameter X_MAX     = 3,//array dimensions
      parameter Y_MAX     = 3,//array dimensions
      parameter X     = 0,    // X address of this node
      parameter Y     = 0,    // Y address of this node
      parameter OP = 0, // 0 = +, 1 = *
      parameter PIPE = 5 // number of pipeline registers
  ) (
      input  wire clk,        // clock
      input wire `D_WIDTH i_operand, //input from Noc
      output reg done_a_pe,   //currently used in mocaarabe to prevent Vivado from treating the modules' as dead code
      output wire `D_WIDTH result
  );
```

**pe_mux**

This module acts as fan-in from torus routers to PE inputs when the number of channels is greater than one, allowing each routing channel's output to go to any one of the PE input ports. It inputs routers' outputs to PE and outputs them to PE inputs. This module acts as a collection of multiplexers, with select lines connected to context memories generated by compiler.

Here is the pe_mux modules' interface:

`pe_mux_3_input.sv` (the 2 input version has 2 data input ports):

```verilog
module pe_mux_3_input #(
    parameter SCHED_LEN = 4,//schedule length
    parameter X = 0,    // X address of this node
    parameter Y = 0     // Y address of this node
) (
    input  wire clk,        // clock
    input  wire rst,        // reset
    input  wire [`BENCHMARK_DATA_WIDTH-1:0] i0, //input
    input  wire [`BENCHMARK_DATA_WIDTH-1:0] i1,
    input  wire [`BENCHMARK_DATA_WIDTH-1:0] i2,
    output reg [`BENCHMARK_DATA_WIDTH-1:0] out0, // output to PE #1
    output reg [`BENCHMARK_DATA_WIDTH-1:0] out1  // output to PE #2
);


```

**torus_switch.sv** 

This module is the NoC router. Each router inputs data from neighboring south and west routers in the same channel and outputs data to its connected PE and neighboring north and east routers. The routing design (what input data goes to each output port) is based on the context memories which act as select lines for north, east, and PE bound muxes. In order to increase operating clock frequency, incoming data is registered and output ports are pipelined as well.

Here are the `tirus_switch` module's interfaces:

```verilog
module torus_switch #(
    parameter SCHED_LEN = 4,
    parameter X_MAX = 3,//array dimensions
    parameter Y_MAX = 3,//array dimensions
    parameter X = 0,    // X address of this node
    parameter Y = 0,        // Y address of this node
    parameter PIPENUM = `TORUS_SWITCH_PIPE_NUM
) (
    input  wire clk,        // clock
    input  wire rst,        // reset
    input  wire `D_WIDTH s_in,  // input data from south
    input  wire `D_WIDTH w_in,  // input data from east
    input  wire `D_WIDTH i_from_pe,     // input data from PE
    output wire `D_WIDTH o_to_pe, // output from this noc node to local PE
    output wire `D_WIDTH n_out, // northbound output data
    output wire `D_WIDTH e_out, // eastbound output data
    output reg done // no data being processed
);
```

Usage
===

To test the design, you would need to run Vivado with an appropriate Tcl script to run the flow and a placement XDC file to do manual placement. To make running the flow easier, we have provided a Tcl script scripts to setup a Vivado run and python scripts to generate placement files and run the flow for an input set of design parameters.

**Requirements**
  - Xilinx Vivado (preferable version 2020.1)
  - Python 3

**placement**: This script generates an XDC constraint file to be used by Vivado. The XDC file targets a clock frequency of 1GHz and makes sure each PE and NoC router is floorplanned properly. In short, each pblock contains a PE, a number of NoC routers equal to the number of communication channels, and fan-in muxes in case they are needed. The added padding around the array and the pblock sizes assure that each block would get the required resources (e.g, DSP blocks) and pblocks do not spill between different Super Logic Regions (SLRs), as they would impose heavy delays on the design. Furthermore, pblocks that need to communicate with other pblocks located at different SLRs are allocated Laguna registers, which allow them to use Super Long Lines (SLLs) for communication between different SLRs. This script is tailored towards Xilinx Alveo U280 board. You can run the script without any input arguments to run the script in it's defauls mode of having x = 19, y = 69 (array dimensions), number of channels = 3, and output file name = `mapping.xdc`. However, you can specify the parameters by running the script with the following parameters in the corresponding order:

From the rtl folder, you can run the following command to generate a placement file:
```bash
python3 placement.py X Y C Output_file_name

Arguments:

  X, Y: aray size dimensions. Valid options are: 1 ≤ X ≤ 19, 1 ≤ Y ≤ 69 
  C: number of communication channels. Valid options are: 1 ≤ C ≤ 3
  Output file name (optional): The name of the output XDC file. Default: mapping.xdc 

```

**Understanding the output of placement generator**

After running the script, you should expect the output placement file to look like this:

```bash
create_clock -period 1.000 -name clk -waveform {0.000 0.500 } [get_ports clk]

#create a pblock for each logick block (PE + NoC routers)
#col number 0

add_cells_to_pblock [get_pblocks y0x0] [get_cells -quiet [list {ys[0].xs[0].cs[0].torus_switch_inst} {ys[0].xs[0].pe_inst}]]
add_cells_to_pblock [get_pblocks y0x0] [get_cells -quiet [list {ys[0].xs[0].mux_inst}]]
add_cells_to_pblock [get_pblocks y0x0] [get_cells -quiet [list {ys[0].xs[0].cs[1].torus_switch_inst}]]
add_cells_to_pblock [get_pblocks y0x0] [get_cells -quiet [list {ys[0].xs[0].cs[2].torus_switch_inst}]]
resize_pblock [get_pblocks y0x0] -add {SLICE_X16Y20:SLICE_X25Y29}
set_property gridtypes {DSP48E2 SLICE} [get_pblocks y0x0]
add_cells_to_pblock [get_pblocks y0x0] [get_cells ys[0].xs[0].pe_inst/* -filter {REF_NAME == DSP48E2}]
.
.
.
#For pblocks connected to pblocks located at other SLRs, use SLLs and laguna registers
create_pblock y10x0
add_cells_to_pblock [get_pblocks y10x0] [get_cells -quiet [list {ys[10].xs[0].cs[0].torus_switch_inst} {ys[10].xs[0].pe_inst}]]

add_cells_to_pblock [get_pblocks y10x0] [get_cells -quiet [list {ys[10].xs[0].cs[1].torus_switch_inst}]]

```

**Running Vivado using cgra.tcl**: The tcl script located at `cgra.tcl` creates a project in Vivado, adds the required files, synthesizes the design using relevant options (like retiming to increase clock frequency), and does placement and routing with various optimizations like phys-opt design, which improves design timing. The resulting design's timing and resource usage after floorplanning are saved in `utilization_report.txt` and `timing_report.txt`. 

The tcl file should look like this:

```tcl
#Create project
create_project cgra-ilp -f ./cgra-ilp -part xcu280-fsvh2892-2L-e
set_property board_part xilinx.com:au280:part0:1.1 [current_project]

#Add files
read_verilog rtl/mocarabe.v
read_verilog rtl/pe_srl.v
read_verilog -sv rtl/torus_switch.sv
read_verilog rtl/mocarabe.h
read_verilog rtl/benchmark.h
read_verilog -sv rtl/pe_2_input.sv
read_verilog -sv rtl/pe_1_input.sv
read_verilog -sv rtl/pe_mux_2_input.sv
read_verilog -sv rtl/pe_mux_3_input.sv
read_verilog rtl/SRL16E.v
read_verilog rtl/SRLC32E.v
read_verilog rtl/SRL64.v

#Read the xdc file
read_xdc mapping.xdc

#Synthesize design
#generics:
# PE_PIPE: Pe pipeline stages (dependant on number of channels)
# PIPENUM: Torous switch output pipeline stages
# PECONFIG: Array configuration. For each PE, 1 = multiplier and 0 = adder. The config bits are input from right to left, with the lsb being the pe at top left, and msb being the pe at bottom right. You can modify it to suit your configuration. Here, 9 is the array size.
synth_design -generic PE_PIPE=7 -generic PIPENUM=1 -generic PECONFIG=36'h101011001 -top mocarabe  -mode out_of_context -keep_equivalent_registers -retiming;
opt_design;
place_design;
phys_opt_design;
phys_opt_design -slr_crossing_opt -tns_cleanup;
route_design;
phys_opt_design;
report_utilization -file utilization_report.txt
report_timing -file timing_report.txt

```

**run_mocarabe**: This script sets up the header files and uses the previous tcl and python scripts to run vivado and synthesize, place, and route the design. Vivado can be set to be run either in batch mode or command line mode. Running vivado in command line enables interacting with the design and Vivado project once florrplanning is done. It is worth noting that for big design, the process can take more than 12 hours.
From the rtl folder, you can run the following command to run the flow:

```bash
python3 run_mocarabe X Y C Batch_mode

Arguments:
  X, Y: aray size dimensions. Valid options are: 1 ≤ X ≤ 19, 1 ≤ Y ≤ 69.
  C: number of communication channels. Valid range: 1 ≤ C ≤ 3.
  Batch_mode: Whether to run Vivado in batch mode (1) or command line mode (0)
```

Running the script without defining any parameters would make it to run in default mode, which gives the following values to the parameters:
 - X = 10
 - Y = 10
 - C = 3
 - batch = 1 


After running the flow, the timing and resource utilization files can be found in this folder. For example, `utilization_report.txt` would contain information about the number of resources (LUT, DSP blocks, etc.) and `timing_report.txt` would contain timing and critical path delay information.
